# Phase 1: Deploy to Vercel — design spec

Date: 2026-05-12
Status: Approved, ready for implementation plan
Predecessor: `2026-05-12-mdx-content-routes-design.md`

## Goal

Ship the existing static Astro build to Vercel under the production domain `hotrod.robotostudio.com`, with the `@astrojs/vercel` adapter installed so that Phase 2 (Vercel Blob + Mux) can layer media optimization on top without further config changes.

## Why Phase 1 exists

- A preview URL per push unlocks faster iteration on content and design.
- Vercel project must exist before Phase 2 can wire Vercel Blob.
- Setting the adapter and domain now means Phase 2 is purely additive — schemas, components, env consumption — not infrastructure plumbing.

## Scope

In scope:

- Add `@astrojs/vercel` as a runtime dependency.
- Update `astro.config.mjs` to use the adapter with `output: 'static'` and `imageService: true`.
- Document the deploy workflow in `README.md`.
- Add `.env.example` (placeholder values only) and harden `.gitignore`.
- Configure the Vercel project (`roboto-pro/hotrod`, already linked) so production builds succeed and serve from `hotrod.robotostudio.com`.

Explicitly out of scope:

- Any change to `src/**` (components, layouts, schemas, content).
- `src/env.ts` Zod-validated env parsing — deferred to Phase 2, which will land it alongside the code that actually reads the variables.
- A `vercel.json` file. Vercel auto-detect is sufficient for now.
- Image optimization usage. The adapter enables it; consumers come in Phase 2.
- RSS, sitemap, OG images, redirects, ignored build steps. Already deferred per `docs/next-up.md`.

## Decisions and reasoning

| Decision | Choice | Why |
|---|---|---|
| Adapter | `@astrojs/vercel` | Future-ready: Phase 2 media work benefits from Vercel image optimization without further config. Cost is one dep + two-line config change. |
| Output mode | `static` | All current routes are known at build time. Server mode would invert the model and add function invocations for no current benefit. |
| Vercel team | `roboto-pro` | Already linked via dashboard. |
| Linking method | Dashboard import + local `vercel link` | Already done by the user. |
| `vercel.json` | None | Auto-detect handles framework preset, install, build, output, Node version (from `engines.node`). No redirects, headers, or ignored-build needs yet. |
| Env var strategy | Vercel-only canonical store; repo gets `.env.example` with placeholder values | Real secrets never live in the repo. `.env.example` exists so a fresh clone (human or agent) sees the expected key names. Zod validation deferred to Phase 2 to avoid validating unused vars. |
| Production domain | `hotrod.robotostudio.com` configured in Phase 1 | Domain is decided; wiring it now costs one DNS record and one click. Avoids a follow-up "ops-only" phase. |
| Image service | `imageService: true` on the adapter | Free upside: any future `<Image>` usage routes through Vercel's CDN. No consumers in Phase 1; the flag is dormant. |

## Architecture

Three layers, each independently verifiable:

1. **Repo configuration.** Adapter installed, `astro.config.mjs` updated, `.gitignore` and `.env.example` added, README documents the deploy workflow.
2. **Vercel project.** Already linked at `roboto-pro/hotrod`. Production branch = `main`. Framework preset = Astro (auto-detected). Node 22 from `engines.node`. Install + build commands auto-detected. Mux env vars (`MUX_TOKEN_ID`, `MUX_TOKEN_SECRET`) added by the user via dashboard or `vercel env add`; Blob token already present.
3. **Production domain.** `hotrod.robotostudio.com` added in Vercel project Domains. DNS: `CNAME hotrod → cname.vercel-dns.com` in the `robotostudio.com` zone. Vercel auto-provisions TLS.

## File changes

Modified:

- `package.json` — add `@astrojs/vercel` to `dependencies` (latest version compatible with Astro 6).
- `pnpm-lock.yaml` — regenerated by `pnpm install`.
- `astro.config.mjs` — import the Vercel adapter; set `output: 'static'`, `adapter: vercel({ imageService: true })`.
- `.gitignore` — add `.env`, `.env.local`, `.env.*.local`, `.vercel/`.
- `README.md` — new "Deploying" section after the content workflow section.

New:

- `.env.example` — three keys (`BLOB_READ_WRITE_TOKEN`, `MUX_TOKEN_ID`, `MUX_TOKEN_SECRET`) with placeholder values and a one-line comment pointing at `pnpm dlx vercel env pull .env.local` as the source of truth.

Not touched:

- `src/**` — no component, layout, schema, or content changes.
- No `vercel.json`.
- No `src/env.ts`.

## Verification

Three gates, in order. If any fails, stop and fix before claiming Phase 1 done.

1. **Local build passes.**
   - `pnpm install` adds the adapter cleanly.
   - `pnpm astro check` passes — types still clean with the adapter imported.
   - `pnpm build` completes, emits the same routes as before, log shows the Vercel adapter active.

2. **First Vercel deploy succeeds.**
   - Push to `main`. Vercel build log shows Node 22, pnpm 10.x, Astro 6, Vercel adapter detected.
   - Auto URL (`hotrod-*.vercel.app`) serves `/`, `/about`, `/services/astro`, `/blog`, `/blog/<slug>`, `/authors`, `/authors/<slug>` — all 200.

3. **Custom domain serves.**
   - After DNS + domain wiring: `hotrod.robotostudio.com` resolves with valid TLS.
   - Same routes return 200; HTTP → HTTPS redirect works.
   - Ad-hoc Playwright smoke check (run by the agent via the Playwright MCP, not a checked-in test) hits the live domain's homepage, one blog post, and the authors index — all 200, no console errors.

## Out-of-band tasks (user, not agent)

- Add `MUX_TOKEN_ID` and `MUX_TOKEN_SECRET` to Vercel for Development, Preview, and Production environments.
- Rotate the Mux secret and Blob token that were pasted into chat during brainstorming.
- Add `CNAME hotrod → cname.vercel-dns.com` at the DNS host that manages `robotostudio.com`.
- Add `hotrod.robotostudio.com` in the Vercel project Domains tab.

## Non-goals

- No framework changes beyond adapter installation.
- No content or schema changes.
- No env-var Zod schema (Phase 2).
- No redirects, sitemap, RSS, OG, search, pagination, tag pages.
- No image transformation pipeline. Adapter enables image optimization; consumers come in Phase 2.

## Risks

- **Adapter version mismatch with Astro 6.** Mitigation: pin to the latest `@astrojs/vercel` that explicitly lists Astro 6 as a peer. If the install warns, the implementation plan stops and reports back.
- **DNS propagation delay.** Mitigation: domain verification is the last gate, not blocking; the auto URL is good enough for Phase 2 start.
- **`.vercel/` already committed before this work.** Mitigation: implementation plan checks `git ls-files .vercel/` and removes any tracked files explicitly.

## Success criteria

Phase 1 is done when:

- The three verification gates pass.
- `hotrod.robotostudio.com` serves the site.
- A fresh clone can run `pnpm dlx vercel env pull .env.local && pnpm dev` and have a working local dev environment.
- `docs/next-up.md` is updated to mark Phase 1 complete and point at Phase 2.
